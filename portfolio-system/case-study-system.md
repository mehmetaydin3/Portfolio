# Case Study System

## Purpose

This document defines the narrative architecture and storytelling framework used to create product design case studies within this portfolio.

It acts as a **design system for portfolio storytelling**, ensuring each project communicates:

• product thinking  
• systems design  
• platform strategy  
• measurable impact  

The goal is to produce case studies comparable to product storytelling used by:

Apple  
Stripe  
Airbnb  
Linear  
Vercel  

Case studies should read like **product narratives**, not UX process documentation.

---

# Core Principles

## Product Thinking

Case studies should demonstrate how the designer helped shape the product strategy.

The narrative should emphasize:

• product opportunities  
• system architecture  
• design decisions  
• behavioral outcomes

Avoid focusing on tools or deliverables.

---

## Systems Thinking

Solutions should be framed as systems rather than individual screens.

Examples include:

• routing systems  
• marketplaces  
• modular content systems  
• subscription platforms  
• personalization engines

Show how the experience scales.

---

## Impact

Every case study must clearly demonstrate measurable outcomes.

Examples:

• engagement increases  
• conversion improvements  
• user behavior shifts  
• operational improvements

Metrics should be tied to design decisions whenever possible.

---

# What Case Studies Should Avoid

Avoid presenting work as a chronological process.

Do NOT structure case studies like:

• first we researched  
• then we ideated  
• then we wireframed  
• then we tested  

Avoid excessive process documentation.

Focus on decisions and outcomes instead.

---

# Narrative Structure

All case studies must follow this structure.

---

# 1. Hero

Purpose:

Introduce the project and its strategic significance.

Content:

Project title  
Short description  
Role  
Timeline  
Platform scope  
Key outcome metrics

Example:

Headline  
Designing a Scalable Engagement Platform

Subheadline  
A modular system that guides users from uncertainty to confident action.

Visual:

Large hero image of the product.

---

# 2. Context

Purpose:

Explain the environment and business landscape.

Answer:

• what the product is  
• who it serves  
• why the work mattered  

Keep this section concise.

Avoid long historical explanations.

---

# 3. Opportunity

Purpose:

Frame the problem as a **product opportunity**.

Example:

Users researching vaccines encountered fragmented and evolving guidance, making it difficult to determine whether recommendations applied to them.

This created an opportunity to design a system that guides users from curiosity to confident decision-making.

Supporting evidence may include:

• research insights  
• user quotes  
• behavioral data

---

# 4. Experience Architecture

Purpose:

Demonstrate systems thinking.

Explain how the experience is structured.

Examples:

• marketplace architecture  
• routing systems  
• modular design systems  
• personalization frameworks  

Visuals should include system diagrams when appropriate.

---

# 5. Product Design Decisions

Purpose:

Highlight the most meaningful design choices.

Each decision should include:

Feature  
Why it mattered  
What changed  
Impact

Example:

### Progressive Disclosure

Complex health inputs were broken into single-question interactions.

Outcome: 87% completion rate.

Focus on decisions that influenced product outcomes.

---

# 6. Feature Spotlight

Purpose:

Provide a deep dive into a meaningful feature.

This section should demonstrate:

• empathy  
• accessibility thinking  
• edge case design  
• craft

Visuals should clearly show the feature in context.

---

# 7. Final Experience

Purpose:

Showcase the finished product.

This section should be **visual heavy**.

Include:

• UI screens  
• flows  
• device mockups  
• responsive layouts

Avoid long text blocks.

---

# 8. Outcomes

Purpose:

Present measurable results.

Examples:

+41% quiz start rate  
87% quiz completion rate  
+42% scroll depth  

Explain what these metrics represent.

Example:

Progressive disclosure reduced friction and increased completion rates.

---

# 9. Reflection

Purpose:

Close the story with mature insight.

Example:

Thoughtful UX architecture shifts behavior not by pushing harder, but by making the next step feel clear, personal, and safe.

Avoid generic reflections.

---

# Writing Style

The writing tone should be:

• concise  
• confident  
• editorial  
• strategic  

Avoid filler phrases such as:

• we realized  
• we discovered  
• the goal was to  
• we decided to  

Instead use direct statements.

Example:

Instead of:

We realized users struggled to interpret eligibility guidance.

Write:

Eligibility guidance was difficult for users to interpret.

---

# Visual Storytelling Principles

Case studies should follow a rhythm:

Text → Visual → Feature → UI → Outcome

Avoid long text blocks.

Visual storytelling should dominate the experience.

---

# Image Preparation Rules

When using presentation materials:

Do not embed slides directly.

Instead:

• crop UI elements  
• remove slide titles  
• rebuild diagrams  
• frame screens cleanly  

Preferred image width:

1200–1600px.

---

# Case Study Quality Checklist

Before publishing confirm:

• the narrative emphasizes product thinking  
• systems thinking is clearly communicated  
• design decisions connect to outcomes  
• visual storytelling dominates the page  
• writing is concise and confident  
• the page does not resemble a slide deck  

---

# How AI Should Use This System

When generating case studies:

1. Follow the narrative structure strictly.
2. Emphasize product architecture and decisions.
3. Avoid process-heavy storytelling.
4. Connect design decisions to measurable outcomes.
5. Maintain a confident editorial tone.

This document should be treated as the governing system for all portfolio case study generation.
